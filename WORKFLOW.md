# 猫掌柜 AI 漫剧创作工作台：跨模型执行手册

版本：2.0  
最后更新：2026-07-16  
适用对象：Codex、Claude、Gemini、GPT 系列及其他能够读文件并调用工具的大模型。

这份文档是项目工作流的操作协议。任何接手模型都应先读完本文件，再开始推进项目。若文档与程序硬门禁不一致，以程序门禁为准，同时记录冲突并修正文档；不得绕过数据库阶段、用户审核或付费确认。

## 1. 不可违背的底线

1. 始终先读取项目当前状态，不凭聊天记忆猜阶段、版本、资产或生成结果。
2. 总导演负责理解、调度、门禁、版本和交接；专业产物由对应角色完成。
3. 创作、审核和用户确认是三件事。创建产物的角色不能在同一工作轮次里自行批准它。
4. 新版本只能新增，不能覆盖已通过版本；所有审核必须绑定当前最新产物 ID。
5. 用户没有在工作台点击真实生成按钮时，任何 Agent 都不得自行调用图片、视频或音频生成接口。
6. 视觉风格未锁定时，禁止真实生图、生视频和豆包音频；不得自动补成港风、90 年代香港电影、胶片或霓虹。
7. 代表性图片和代表性视频没有通过前，禁止批量生成。
8. 用户拒绝的视频不能更新连续性，也不能成为后续镜头的来源；只有已接受视频的真实尾帧和真实结束状态才是下一镜头依据。
9. 用户退回时只修改受影响对象；不得无理由重做全部剧本、资产或镜头。
10. 不直接修改数据库来跳阶段，不伪造审核、生成完成、费用、媒体文件或 Skill 验证结果。

## 2. 新模型接手后的启动协议

按以下顺序执行，不得跳步：

1. 阅读根目录 `AGENTS.md`、本文件和当前任务明确点名的 Skill。
2. 运行 `get_skill_status`，确认四个项目级 Skill 最近一次验证状态。
3. 已知项目 ID 时运行 `get_project_context`；未知时先运行 `list_projects`。只有用户明确要求新建项目时才运行 `create_project`。
4. 读取当前 `stage`、内容模式、视觉风格状态、最新产物版本、未解决退回项、资产、分镜和生成任务。
5. 判断任务属于剧情短片、创意广告或音乐 MV，并加载 `creative-production-orchestration`。
6. 根据当前阶段只调用该阶段的专业角色和 Skill，不提前生产后续产物。
7. 任何写入后再次读取项目上下文，确认阶段和版本确实变化，再向用户报告。

如果用户说“处理工作台最新退回项”，先运行 `list_open_revisions`。如果用户说“处理工作台待生图任务”，先运行 `list_pending_codex_image_requests`，不得让用户重复粘贴工作台已有信息。

## 3. 运行环境兼容规则

### 3.1 有子 Agent 和 MCP 的环境

优先调用 `.codex/agents` 中的专业 Agent。每次委派必须提供：项目 ID、当前阶段、最新锁定输入、待解决问题、输出格式和禁止越界事项。专业 Agent 返回后，总导演只做契约检查与流程推进。

### 3.2 没有子 Agent、但能读文件的大模型

允许同一个模型按“隔离角色轮次”顺序执行，但必须满足：

- 每次只扮演一个角色，先读取该角色 TOML 和相关 Skill。
- 在产物开头记录 `role`、输入版本 ID、输出类型和时间。
- 创作轮次结束后清空创作立场，再以独立审核角色检查；不能用“我刚写的所以我认为通过”代替评分证据。
- 总导演、导演审核、观众审核必须保存为三个独立产物/审核记录。
- 如果运行环境不能形成独立审核轮次，停止在待审核状态，请用户或另一模型接手，不能自批。

### 3.3 没有 MCP 的环境

不得假装已写入工作台。可以按照本文件生成结构化产物，但必须明确标记“尚未写入工作台”。若能访问本地工作台 HTTP 接口，可使用非付费接口写入；真实生成仍只能由用户在界面点击。禁止直接改 SQLite 数据库。

### 3.4 Agent 配置更新后

Agent TOML 在任务启动时加载。修改 `.codex/agents` 后，应重新打开一个任务或重启相应 Agent 运行时，再验证所有角色可用；当前已打开任务里看不到新角色，不等于配置仍然错误。

## 4. 内容模式与创作档案

每个项目在写剧本前必须锁定以下字段：

- `contentMode`：`short_film`、`ad`、`mv` 三选一。
- 目标受众。
- 目标平台和画幅；抖音默认 9:16，但平台不决定视觉风格。
- 目标时长。
- 情绪曲线。
- 传播、转化或表达目的。
- 预算意识与样片策略。
- 验收标准。
- `visualStyle.status`：`locked` 或 `needs_review`。

模式分流：

| 模式 | 剧本与审核重点 | 必须增加的专业检查 |
| --- | --- | --- |
| `short_film` | 人物目标、阻力、变化、结果、情绪推进和余味 | 导演、观众、摄影、声音、剪辑 |
| `ad` | 痛点、产品价值、证据、品牌记忆、自然 CTA | 上述全部，加 `brand-reviewer` |
| `mv` | 音乐段落、歌词情绪、视觉意象、高潮和复看价值 | 声音与剪辑重点检查 BPM、同步、高潮和音乐版权 |

视觉风格只有三种合法来源：剧本中的明确可执行证据、用户明确确认、已锁定风格资产。只有“雨夜”“霓虹”“城市”等零散词不等于港风。用户或风格资产已经锁定的视觉风格，剧本修订时不得被自动推断覆盖。

## 5. 专业 Agent 责任边界

| Agent | 只负责什么 | 主要输出 | 不能做什么 |
| --- | --- | --- | --- |
| `creative-producer` | 把 brief 变成创作档案 | 内容模式、受众、平台、时长、目的、预算、验收标准 | 不写完整剧本，不擅自定风格 |
| `screenwriter` | 写可拍、可视化、可拆镜头的剧本 | 场次、目标、阻力、变化、结果、对白和声音线索 | 不自审通过，不写最终模型提示词 |
| `director-reviewer` | 审戏剧核心、电影感、节奏和可拍性 | 评分、通过/退回、责任对象修改项 | 不改写并同时批准自己的版本 |
| `audience-reviewer` | 审前三秒、理解成本、情绪和完播动力 | 评分、受众风险、具体修改项 | 没真实数据时不得伪装成平台结论 |
| `cinematographer` | 设计景别、构图、运动、焦点、光色和连续性 | 摄影方案 | 不改剧情，不输出最终模型提示词 |
| `asset-designer` | 建立角色、服装状态、场景、道具、风格资产 | 资产清单、身份锚点、参考图职责 | 不越级写最终生图提示词 |
| `asset-continuity-reviewer` | 检查跨镜头身份与空间连续性 | 漂移清单、锁定项 | 不重新设计全部资产 |
| `storyboard-artist` | 把锁定剧本拆成可生产镜头 | 完整镜头表与状态交接 | 不自定未锁定风格 |
| `image-prompt-designer` | 使用图像 Skill 编写/检查最终生图提示词 | 最终提示词、参考图职责、禁止变化项、lint | 不提交付费任务，不把资产 ID 写入提示词 |
| `video-prompt-designer` | 使用 Seedance Skill 编写/检查最终视频提示词 | 自然语言视频提示词、连续性状态、lint | 不凭计划状态覆盖真实尾帧 |
| `audio-supervisor` | 规划声线、对白、环境声、音乐、音效和字幕 | 豆包纯文本提示词、音频资产方案、lint | 不模仿具体演员或未授权原声 |
| `editor-reviewer` | 审节奏、衔接、字幕、混音和导出 | 按责任对象分类的修复项 | 不只给“节奏不好”一类空话 |
| `brand-reviewer` | 广告真实性、禁用表达、品牌与版权 | 风险等级、证据、改法 | 不编造产品数据或法律结论 |
| `generation-supervisor` | 定位退回原因并定向返工 | 问题分类、责任对象、最小修改范围 | 不默认全量重做 |

总导演不得吞并专业角色。跨模型兼容时采用第 3.2 节的隔离角色轮次，不等于取消角色边界。

## 6. 四个项目级 Skill 的强制路由

### `creative-production-orchestration`

所有项目入口都先使用。它决定内容模式、专业角色组合、内部审核和用户验收顺序。

### `gpt-image-2-storyboard`

所有最终资产图、分镜首帧、多参考图融合和局部重绘提示词都必须使用。

角色设定图硬规范：

- 3:2 横向。
- 左侧约 40%：同一角色脸部特写。
- 右侧约 60%：正面全身、侧面全身、背面全身三视图。
- 浅灰中性背景、柔和均匀棚光，不使用剧情光效。
- 明确左右同一角色，脸型、五官、发型、体型、服装和配饰不得漂移。
- 标准参数：`gpt-image-2-official`、1K、high、单张。

资产 ID、项目名、版本号只放结构化字段，不能进入发给模型的自然语言提示词。提交前运行：

```powershell
python .agents/skills/gpt-image-2-storyboard/scripts/prompt_lint.py --prompt-file <提示词文件> --aspect <画幅>
```

### `doubao-audio-generation`

所有豆包音频、角色音色母版、场景对白母带和镜头音频切片都必须使用。提示词固定顺序：角色声线清单 → 环境/音乐 → 按时间顺序的对白与音效。只有剧本明确是 90 年代香港电影时才使用港式普通话。

```powershell
python .agents/skills/doubao-audio-generation/scripts/prompt_lint.py <提示词文件> --style auto
```

明确的 90 年代港片项目才可把 `auto` 改为 `hk90`。

### `seedance-20`

所有最终视频提示词、连续镜头、参考音视频和视频返工都必须使用。项目始终按 `sequence_project` 管理；单镜头允许 4—15 秒。最终表面提示词使用清楚的自然语言，不把工作台 JSON 或管理字段直接塞给模型。

连续镜头必须记录：关系类型、计划起始状态、计划结束状态、已经发生、留给后续、连续性锁定、允许变化、真实结束状态和真实尾帧。

## 7. 十二阶段状态机

只能按程序允许的方向推进。每次进入下一阶段前检查退出条件。

| 阶段 | 主责任 | 必须产物/动作 | 退出条件与下一阶段 |
| --- | --- | --- | --- |
| `idea` | 总导演 + `creative-producer` | 创作档案、视觉风格状态 | 档案足够，`screenwriter` 写剧本并保存 → `script_internal_review` |
| `script_internal_review` | `director-reviewer` → `audience-reviewer` | 两份绑定当前剧本 ID 的审核 | 两项均达标 → `script_user_review`；退回仍在本阶段 |
| `script_user_review` | 用户 | 审最新剧本 | 通过并锁定 → `asset_design`；退回 → `script_internal_review` |
| `asset_design` | `asset-designer` + `image-prompt-designer` | 角色、场景、道具、风格资产与合格提示词 | 资产方案保存 → `asset_user_review` |
| `asset_user_review` | 用户 + `asset-continuity-reviewer` | 每个资产逐项看图、通过或退回 | 所有资产逐项锁定后整体通过 → `storyboard_design` |
| `storyboard_design` | `cinematographer` + `storyboard-artist` + 提示词/声音角色 | 完整镜头表、首帧提示词、视频提示词、声音方案 | 分镜版本保存 → `storyboard_user_review` |
| `storyboard_user_review` | 用户 | 审全部镜头 | 通过 → `sample_image`；已有锁定首帧时可到 `sample_video`；退回 → `storyboard_design` |
| `sample_image` | 用户 + `generation-supervisor` | 一个代表性镜头的合格首帧 | 用户锁定候选图 → `sample_video` |
| `sample_video` | 用户 + `generation-supervisor` | 同一代表镜头的视频、真实尾帧和结束状态 | 用户通过 → `batch_generation` |
| `batch_generation` | 总导演调度，各专业角色复核 | 所有镜头图片/视频/声音完成并逐项通过 | 全部镜头视频通过后生成本地预览 → `final_review` |
| `final_review` | `editor-reviewer`，广告加 `brand-reviewer`，最后由用户验收 | 成片预览、修复清单、最终验收 | 用户通过 → `completed`；退回 → `batch_generation` |
| `completed` | 总导演 | 锁定最终版本和交付说明 | 流程结束；新改动必须开新版本 |

### 内部审核门禁

- 导演和观众审核都必须绑定“当前最新剧本”的同一个 artifact ID。
- 观众审核前，该剧本版本必须已有导演通过记录；旧剧本的导演通过不能沿用。
- 通过标准：平均分不低于 4，任何关键项不低于 3。
- 内部自动返工最多 3 轮。第三轮仍不通过时停止自动推进，记录退回项，请用户裁决。

### 资产门禁

- 资产必须逐项通过，不能只点一次整体通过。
- 修改资产只让引用它的镜头失效。
- 角色资产图不符合 3:2 左脸/右三视图结构时，服务端禁止创建真实任务。
- 新生成会使旧批准失效；Codex 待生图任务未完成时不能锁定旧图。

### 分镜与连续性门禁

- 每个镜头必须有叙事目的、4—15 秒时长、画面构图、动作、对白/声音、资产引用、计划起止状态和连续性字段。
- 普通镜头先锁定首帧才能生视频。
- `seamless_continuation` 必须指定上一镜头；上一镜头未通过、无真实结束状态或无尾帧时，后续镜头不得生成。
- 视频通过时必须记录真实结束状态并保存尾帧；计划状态不能替代观察结果。

## 8. 版本、退回与定向返工

### 版本规则

- 剧本、资产方案、分镜和最终导出都通过新版本保存。
- 新剧本会使旧内部审核失去推进资格。
- 用户或风格资产锁定的视觉风格在剧本修订时继续保留。
- 已锁定版本不覆盖；需要调整时创建 V2、V3。

### 退回处理

1. 读取未解决退回项。
2. `generation-supervisor` 将问题分成：提示词、资产、模型/参数、动作/连续性、声音、后期/平台。
3. 指定唯一主要责任对象和受影响 ID。
4. 保留已经正确的部分，只改一个主要变量。
5. 保存新版本或新生成候选。
6. 由原审核角色复核；通过后再解决退回项。

常见映射：脸和服装漂移 → 资产；空间关系错误 → 场景/分镜；动作过多或镜头漂移 → 视频提示词；口型和音色 → 声音；字幕安全区和节奏 → 剪辑。

## 9. 真实生成与费用权限

### APIMart 图片/视频与火山音频

这些是付费或可能消耗额度的真实生成，只能由用户在工作台界面点击按钮触发。MCP 不提供绕过按钮的付费提交能力。Agent 可以准备和检查提示词、参数和资产引用，但不能替用户点击、代提交或批量开启。

### Codex 内置图片通道

只有用户先在工作台创建了 Codex 生图请求后，当前对话才能处理：

1. `list_pending_codex_image_requests` 读取队列。
2. `claim_codex_image_request` 领取一项。
3. 使用内置图像生成能力，严格使用请求中的提示词、画幅和参考图。
4. 成功后用 `complete_codex_image_request` 导入本地图片。
5. 失败后用 `fail_codex_image_request` 写回真实错误，不伪造结果。

同一资产或镜头同一时间只能有一个生图通道。任务未完成、失败或取消前，不能在 Codex 与 APIMart 之间重复提交。

### Mock 的真实边界

Mock 只能验证接口外壳和部分非生成流程，不能在资产中心或分镜台冒充真实图片/视频，也不能证明视觉工作流已完整跑通。需要全流程视觉验收时，必须使用用户已授权的 Codex 图片通道或付费通道。

## 10. 工作台 MCP 工具清单

以下是当前实际注册的全部工具。工具名变更时必须同步修改本文件和契约校验器。

| 工具 | 用途 | 使用限制 |
| --- | --- | --- |
| `list_projects` | 列出项目 | 只读 |
| `create_project` | 新建项目 | 仅用户明确要求时 |
| `set_creative_profile` | 设置内容模式、平台和视觉风格 | 风格无证据时使用 `needs_review` |
| `get_project_context` | 读取完整项目上下文 | 每次接手和写入后都应使用 |
| `save_artifact_version` | 保存创意、剧本、审核报告、资产方案、分镜或导出版本 | 只新增版本，不覆盖锁定版本 |
| `submit_internal_review` | 提交导演或观众内部审核 | 必须绑定当前剧本 ID，遵守评分和先后顺序 |
| `upsert_asset` | 新增或修改资产 | 修改后只使引用镜头失效 |
| `upsert_shot` | 新增或修改镜头 | 4—15 秒，最终提示词先过相应 Skill |
| `list_open_revisions` | 读取未解决退回项 | 处理退回前先读 |
| `list_pending_codex_image_requests` | 读取用户已创建的 Codex 生图请求 | 不能自行创建付费任务 |
| `claim_codex_image_request` | 领取一个 Codex 生图请求 | 领取后必须完成或写回失败 |
| `complete_codex_image_request` | 导回真实生成图片 | 文件必须真实存在并属于本次请求 |
| `fail_codex_image_request` | 写回生成失败 | 保留真实错误原因 |
| `resolve_revision` | 解决已完成的退回项 | 只有修改并复核后才能调用 |
| `get_skill_status` | 查看项目级 Skill 验证状态 | 状态过期或失败时先停止生产并重新验证 |

用户剧本、资产、分镜和最终成片的“通过/退回”在工作台界面完成，不由内部审核工具代替。真实生成、媒体锁定、视频验收和本地预览也由工作台界面及其服务端门禁处理。

## 11. 每次交接必须携带的状态

换 Agent、换模型、换任务或隔天继续时，交接至少包含：

- 项目 ID、内容模式、平台、画幅、目标时长。
- 当前阶段。
- 视觉风格状态、来源和证据。
- 当前最新剧本/资产方案/分镜版本 ID 与锁定状态。
- 已通过和待审核的门禁。
- 未解决退回项及受影响对象。
- 已锁定角色、场景、道具、风格资产 ID。
- 每个镜头的关系类型、计划/真实结束状态、尾帧和声音状态。
- 正在运行、失败或等待 Codex 的生成任务。
- 下一步唯一允许动作。

推荐交接句式：

```text
项目 {projectId} 当前处于 {stage}。最新剧本为 {artifactId/version}，视觉风格 {locked/needs_review}，未解决退回项 {数量与ID}。已锁定资产 {列表}。当前只能执行 {下一步动作}；禁止 {越级动作或真实生成限制}。
```

## 12. 验证、更新和故障处理

### 一次性全量验证

```powershell
npm run typecheck
npm test
npm run build
npm run validate:skills
```

`validate:skills` 会检查四个项目级 Skill、Seedance 上游完整测试套件、14 个 Agent 的 TOML 和跨模型配置、Agent 责任登记、MCP 工具注册表以及本文件中的工具名。

### Skill 更新规则

- `seedance-20`：只通过固定上游来源同步，记录版本、commit、校验和和完整测试结果。
- `gpt-image-2-storyboard` 与 `doubao-audio-generation`：模型与接口规则优先采用官方文档；更新后必须增加正反例测试。
- `creative-production-orchestration`：外部 GitHub 项目只提供方法参考，必须适配本项目状态机和审核门禁。
- 所有外部来源、许可证和本地落点登记在 `docs/external-sources.md`。

### 出现冲突时

1. 先停止当前阶段写入和真实生成。
2. 读取项目上下文和错误信息，确认是文档、Agent、Skill、接口还是数据状态冲突。
3. 以程序硬门禁和最新锁定用户决定为准。
4. 修复后运行相关单测，再运行全量验证。
5. 更新本文件和来源登记，避免下一个模型重复踩坑。

### 完成报告最低要求

向用户说明：完成了什么、当前阶段、哪些测试通过、是否产生真实费用、仍有什么风险、下一步唯一建议。不得只说“已经优化”而没有可验证结果。
